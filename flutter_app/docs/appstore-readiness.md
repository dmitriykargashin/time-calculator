## Verdict
Yes — this app is approvable: a free, fully offline time calculator with voluntary developer-tip IAPs is an explicitly permitted model (Guideline 3.1.1 "tip the developer"), the toolchain already meets the April 2026 Xcode 26/iOS 26 SDK upload mandate, and the privacy story (Data Not Collected, no ATT, plugin privacy manifests all present) is clean. What stands in the way is that the iOS side has never actually been built or branded: the app icon is still the Flutter template placeholder (with no usable 1024px source art), the Support screen ships four dead purchase buttons on iOS, the rate fallback and share text link to Google Play, signing/CocoaPods are unconfigured, and the custom star-rating dialog with review gating violates the verbatim 5.6.1 ban on custom review prompts. All of it is roughly a day of code work plus App Store Connect paperwork; the only genuine judgment risk afterward is the June 9, 2026 Guideline 4.3 tightening against "simple timer"-adjacent utilities, mitigated by leading with the app's differentiated features in metadata.

## Blockers (would cause rejection/upload failure)
- Placeholder app icon (Guideline 2.3.8 / 2.1(a)): every PNG in ios/Runner/Assets.xcassets/AppIcon.appiconset/ is byte-identical (MD5-verified, 1024px = c785f8932297af4acd5f5ccb7630f01c) to the Flutter template 'F' icon, and assets/icon_source.png is only 192x192 — too small to generate the mandatory 1024x1024 sRGB no-alpha marketing icon. Fix: obtain/export the real Cardamon artwork at >=1024px (the Android mipmaps got real icons, so source art exists somewhere) and regenerate the full appiconset.
- Dead purchase UI on iOS (Guideline 2.1 App Completeness): lib/services/monetization.dart:68-69 gates billing to Android only and buy() silently no-ops at :167, while lib/ui/support_screen.dart still renders four enabled 'buy N Cup(s)' buttons reachable from the always-badged tea icon on the main screen. A reviewer will tap a buy button, nothing will happen, and the app gets rejected. Fix: EITHER create the support_1/3/5/9 non-consumables in App Store Connect, widen _isActivePlatform to include TargetPlatform.iOS, and submit the IAPs together with the first binary; OR hide the buy rows / whole tea section (and the toolbar badge) on iOS for v1.
- Google Play link launches inside the iOS app (Guideline 2.3.10): the final fallback in lib/services/rate_service.dart (_openStoreReview, ~lines 192-196) calls launchUrl(kPlayStoreUrl) unconditionally — only the openStoreListing branch at :183 is Android-gated — so an iOS user/reviewer hitting the fallback lands on play.google.com. lib/config.dart:21's share text also embeds https://bit.ly/TimeCalcCardamon (Play listing). Fix: gate the Play URL to Android; once the App Store Connect record exists, use https://apps.apple.com/app/id<APPSTOREID>?action=write-review as the iOS fallback and a platform-conditional store link in the share text.
- Build/upload is currently impossible: no DEVELOPMENT_TEAM anywhere in ios/Runner.xcodeproj/project.pbxproj (legacy 'iPhone Developer' CODE_SIGN_IDENTITY on Runner configs), CocoaPods have never been installed (no ios/Pods, no Podfile.lock), and ios/Podfile line 2 still has `# platform :ios, '13.0'` commented out. Fix: uncomment the platform line, run flutter pub get + pod install, set the team with automatic signing, and do a real device build/archive before anything else — the iOS target has plainly never been exercised.
- Missing 'Restore Purchases' control if iOS IAP ships (Guideline 3.1.1: 'make sure you have a restore mechanism for any restorable in-app purchases'): non-consumables require a user-visible restore path; today restore only happens implicitly inside Monetization.refresh() (monetization.dart:149-151), which on iOS maps to AppStore.sync() and can throw an App Store sign-in prompt at cold launch — the opposite of Apple's documented pattern. Fix: add a visible Restore Purchases button on the Support screen and make the automatic restorePurchases() call Android-only. (Moot only if the Support screen is hidden on iOS for v1.)

## Required App Store Connect / account setup
- Create the App Store Connect app record for bundle id com.cardamon.timeCalculator (note it deliberately differs in casing from the Android applicationId — don't conflate them). This yields the Apple app ID that the rate-fallback, write-review deep link, and share-text code changes depend on, so do it first.
- Sign the Paid Applications Agreement and complete Tax + Banking in App Store Connect BEFORE creating IAP products — sandbox returns zero products until the agreement is active. Then create the four non-consumable products (support_1, support_3, support_5, support_9) with localized names/prices and submit them for review together with the first app version.
- Enroll in the App Store Small Business Program (not automatic — apply before sales): trivially eligible, drops Apple's cut on every tip from 30% to 15%.
- Declare and verify EU DSA trader status (required since Feb 17, 2025, because IAP monetization makes the account a trader; verified address/phone will be displayed on EU storefronts) — otherwise the app is simply unavailable in the EU. Decide consciously if that trade-off is acceptable.
- Write and host a privacy policy (static page is fine): app collects/transmits no data, settings stay on device in local preferences, tips are processed entirely by Apple/StoreKit, feedback email is user-initiated, deleting the app deletes all data, developer contact. Enter the URL in App Store Connect metadata — mandatory under 5.1.1(i) even for zero-collection apps.
- Complete the App Privacy nutrition label: answer 'No, we do not collect data from this app' -> 'Data Not Collected' badge. Do NOT declare Purchase History (StoreKit purchases are Apple's collection; the on-device shared_preferences cache is not 'collection'). Answer 'no' to tracking. Mandatory before first submission.
- Complete the new age-rating questionnaire (the system that replaced the old one as of Jan 31, 2026 — new apps just answer the new questions at setup).
- Prepare screenshots for BOTH device families: one 6.9-inch iPhone set AND one 13-inch iPad set, because TARGETED_DEVICE_FAMILY = "1,2" makes the app universal and review will exercise iPad (alternatively switch to iPhone-only first and decide Apple Vision Pro availability in App Store Connect).
- Fill App Review notes and store metadata leading with the differentiators — multi-unit time arithmetic, selectable result formats, amount-per-interval (salary/distance) calculation, fully offline/no-tracking — and avoid the words 'timer', 'donation', or 'charity'. This is the cheap insurance against the June 9, 2026 Guideline 4.3 tightening and 4.2.

## Judgment-call risks
- Review gating / custom star dialog (HIGH likelihood if shipped, must-fix in practice): one report claimed the 'use the provided API... we will disallow custom review prompts' rule was removed — that is wrong; it is verified verbatim in the current official guidelines as 5.6.1 (renumbered from 1.1.7 in June 2021), and Section 3's 'filtered... feedback' manipulation language plus 5.6.3 Discovery Fraud independently cover the >=4-to-store/<4-to-email split. Enforcement is uneven (many shipped apps gate), but the Support screen's 'Leave a review' button fires the custom dialog on demand, so a reviewer can hit it in seconds. Mitigation: code change #6 above. The same pattern also verbatim violates Google Play's In-App Review rules ('shouldn't ask the user any questions before... presenting the rating card'), so fix Android too.
- Guideline 4.3(b) spam / June 9, 2026 tightening (MEDIUM likelihood): Apple now explicitly refuses new 'simple timer'-type submissions 'unless they offer a meaningfully different or improved experience', and simple calculators are the canonical 4.2 example. A time calculator from a first-time iOS submitter is adjacent to that bucket. Mitigation: metadata/screenshots/review-notes that lead with multi-unit time arithmetic, result formats, and the amount-per-interval salary/distance calculator; expect possible back-and-forth (one appeal per rejection); a platform-leveraging extra (widget/Siri shortcut) would help but isn't required.
- Donation shape (LOW likelihood): four one-time non-consumable tiers with a persistent green star is a coherent, review-safe shape — community precedent shows non-consumable tips approve cleanly while consumable duplicates have drawn odd '3.1.1 - new IAP type' rejections, so keep non-consumables despite one report floating consumables. Known limitation: each tier can only be bought once. Do NOT add an external Buy-Me-a-Coffee/PayPal link (documented 3.1.1 rejection outside the US) and do not pursue US/EU external-purchase links — the US regime is under active Supreme Court challenge and the EU entitlement's stacked fees exceed the SBP 15% IAP rate for sub-$10 tips.
- iPad review surface (LOW likelihood): TARGETED_DEVICE_FAMILY "1,2" means review happens on iPad and 13-inch screenshots are mandatory; Flutter resizes fine and Info.plist already declares all four iPad orientations (satisfying iPadOS 26 windowing now that UIRequiresFullScreen is ignored), but if iPad layouts haven't been eyeballed, either test them or set device family to 1.
- Post-launch horizon (no action now): apps built with the iOS 27 SDK (mandatory ~April 2027) will not launch without the UIScene lifecycle — plan the Flutter UISceneDelegate migration with next year's toolchain bump. Also keep in_app_review current (SKStoreReviewController is deprecated since iOS 18 in favor of AppStore.requestReview; handled inside the plugin).

## Already compliant
- Toolchain already satisfies the April 28, 2026 upload mandate: Xcode 26.5 (iOS 26.5 SDK) installed and Flutter 3.38.5 stable officially supports it; IPHONEOS_DEPLOYMENT_TARGET 13.0 is coherent across all Runner configs and AppFrameworkInfo.plist, and in_app_purchase_storekit's iOS 12 floor is below it. No Flutter upgrade needed.
- Privacy manifests / ITMS-91053 / ITMS-91061: nothing to add. Flutter 3.38.5's engine framework ships PrivacyInfo.xcprivacy (FileTimestamp, SystemBootTime reasons), and all five locked iOS plugin implementations ship manifests (shared_preferences_foundation 2.5.6 covers the UserDefaults required-reason API; in_app_purchase_storekit 0.4.10, url_launcher_ios 6.4.1, in_app_review 2.0.12, share_plus 13.1.0). Runner's own code uses no required-reason APIs.
- Privacy posture overall: no accounts, no network features, no analytics, no ads, no tracking — qualifies for the 'Data Not Collected' label; no ATT prompt or NSUserTrackingUsageDescription needed (adding one would be wrong); Info.plist is clean of ad/tracking keys (no GADApplicationIdentifier, SKAdNetworkItems).
- The monetization model itself: voluntary developer-tip IAPs in a fully functional free app are explicitly blessed by 3.1.1; this is not charity fundraising (3.2.2(iv) inapplicable); nothing is locked behind purchase or rating (3.2.2(x) satisfied); the existing 'support me, as a developer... buy me some cups of tea' copy is exactly the right framing.
- Launch screen requirement: UILaunchStoryboardName = LaunchScreen with Base.lproj/LaunchScreen.storyboard present satisfies the storyboard mandate (the blank-white look is cosmetic only).
- Bundle identity and versioning: com.cardamon.timeCalculator, display name 'Time Calculator Cardamon' (under the 30-char limit), version 2.0.5+20 wired via FLUTTER_BUILD_NAME/NUMBER — all valid for a first upload (just keep feedback_service.dart's hardcoded versionCode 20 in sync).
- mailto feedback path: feedback_service.dart uses launchUrl() directly (never canLaunchUrl), so the absent LSApplicationQueriesSchemes is harmless.
- Bitcode and dSYMs: non-issues on Flutter 3.38 (bitcode removed since Xcode 14; flutter build ipa bundles dSYMs and the historical Flutter.framework dSYM bug is fixed).
- Accessibility Nutrition Labels: still voluntary with no announced deadline — worth filling in early for a simple app, but not blocking.
